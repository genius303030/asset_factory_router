# Owner Pricing Production Import Command Contract

G1-034 defines the design-only command contract for the future owner-pricing
final application path. It does not implement the command, does not register a
parser, does not add command-line flags to code, does not change runtime
behavior, and does not touch real owner data.

## Contract Boundary

The reserved future command label remains:

```text
asset-factory owner-pricing-final-import
```

That command must stay absent from executable code until a separate G0-approved
implementation issue changes the safety gate contract. This document only
defines the future contract that G2 and G0 must review before implementation.

## Required Input Slots

A future implementation must require these logical input slots:

| Slot | Purpose | Required Validation |
| --- | --- | --- |
| Approved sandbox output | The exact sandbox output reviewed by owner/G0. | Exists, parses, checksum matches approval record, sandbox-only markers remain false for production mutation. |
| Owner approval record | The owner approval artifact generated by the approval gate. | Exists, parses, approval type matches, references the intended sandbox output, checksum matches sandbox bytes. |
| Preflight evidence | The read-only final-import preflight report. | Exists, is current for the same artifact packet, status is PASS. |
| Production target | The explicit pricing target to protect and update in the future implementation. | Path is explicit, allowed, expected format, and readable before write. |
| Backup output | The explicit backup destination. | Path is explicit, allowed, absent unless a future overwrite design is approved. |
| Audit log | The explicit audit destination. | Path is explicit, allowed, writable before backup or production write. |
| G0 approval artifact | The final owner/G0 approval packet. | References implementation issue, PR, artifact checksums, G2 review, CI status, rollback plan, and final decision. |
| Final confirmation artifact | The last intentional confirmation for the exact artifact packet. | Present, exact for this attempt, and separate from sandbox approval. |

Exact option names, parser wiring, help text, and shell examples remain blocked
until a future implementation issue. The future command must reject partial
input packets.

## Required Output Slots

A future implementation must produce these outputs:

| Output | Required Behavior |
| --- | --- |
| Backup artifact | Written before production write from the exact pre-import target bytes. |
| Audit log | Records attempt state before backup, after backup, before production write, after production write, and after rollback if needed. |
| Owner report | Summarizes result, artifact paths, checksums, counts, validation results, rollback status, and next safe step. |
| Metrics update | Adds or schedules the PR metrics row with durable evidence. |
| Fixture/golden report updates | Adds fake-only examples for changed command behavior. |

Real owner artifacts must stay in ignored local paths.

## Required Validation Order

The future command must run in this order:

1. Parse the approved argument slots.
2. Confirm the repository state is clean enough for the approved operation.
3. Validate sandbox output.
4. Validate approval record.
5. Match approval checksum to sandbox output bytes.
6. Validate preflight evidence.
7. Validate G0 approval artifact.
8. Validate production target path and current bytes.
9. Validate backup output path.
10. Validate audit log path and write initial audit state.
11. Create backup from production target bytes.
12. Verify backup checksum equals pre-import production checksum.
13. Write production target from approved sandbox output.
14. Validate production target schema, counts, checksums, and duplicate keys.
15. Write owner report and final audit state.

Any failed step before production write must abort with no production write. Any
failed step after production write must enter rollback review.

## Required Fail-Closed Errors

The future implementation must provide clear errors for:

- Missing sandbox output.
- Sandbox output parse failure.
- Sandbox output is not sandbox-only.
- Missing approval record.
- Approval record parse failure.
- Approval record type mismatch.
- Approval checksum mismatch.
- Stale approval record.
- Missing preflight evidence.
- Failing preflight evidence.
- Missing G0 approval artifact.
- G0 approval artifact does not match this attempt.
- G2 review is missing or request-changes.
- Unsafe production target path.
- Missing production target.
- Production target format mismatch.
- Unsafe backup path.
- Backup output already exists.
- Backup write failure.
- Backup checksum mismatch.
- Audit log path unsafe or unwritable.
- Duplicate material keys.
- Skipped invalid rows in importable materials.
- Unrelated dirty repository state.
- Owner/G0 approval withdrawn.
- Post-import validation failure.
- Owner report write failure.
- Rollback verification failure.

Each error must state the failed gate, whether production write happened, and
the next safe step.

## Required Safety Invariants

A future implementation must preserve these invariants:

- No production write before verified backup.
- No production write before audit state is writable.
- No production write before G0 approval artifact is present.
- No production write before G2 review is PASS.
- No production write when checksums mismatch.
- No production write when fake or sandbox-only markers are inconsistent.
- No retry in place after a production write failure.
- No real owner pricing data committed to Git.
- No silent overwrite of production target, backup, audit, report, fixture, or
  metrics artifacts.

## Future CI Contract

Before implementation, the safety gate must be redesigned from "reserved command
must be absent" to "approved command must satisfy the reviewed contract." That
redesign must be its own reviewable task and must keep the current checks for:

- Real owner pricing data not being tracked.
- Private/output paths being ignored.
- Owner-pricing unit tests running in CI.
- Safety gate running in CI.
- Metrics validation running before review.

The first implementation PR must not merge unless the updated safety gate proves
every required fail-closed case.

## Documentation, Fixture, And Metrics Contract

The implementation PR must update:

- Command documentation.
- G0 approval checklist.
- G2 review checklist.
- Fake fixture registry.
- Golden report policy.
- Metrics baseline row or a committed follow-up note.

Use `unknown` for metrics values that are not yet visible. Do not invent test
counts, CI status, review status, or production behavior status.

## G0 Approval Gate

G0 must approve the future implementation issue before any parser or runtime
code is added. G0 approval must name:

- Implementation issue.
- Implementation PR.
- Reviewed command contract version.
- Artifact packet.
- G2 review result.
- CI evidence.
- Rollback plan.
- Final decision.

Without this approval packet, implementation remains blocked.
